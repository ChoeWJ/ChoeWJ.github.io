# 👑 로그 설계를 위한 데이터 기초 지식

## 🍡 데이터 로그의 종류

- 데이터 로그는 크게 서비스 로그와 사용자 행동 로그로 나눌 수 있다.
- 서비스 로그는 데이터베이스에 저장되어 주로 운영 데이터를 의미하며, 예시로 고객의 가입 날짜나 구매 내역 등이 있다.
- 사용자 행동 로그는 유저의 클릭이나 화면 이동과 같은 활동을 기록하여 분석 목적에 사용되는데 이 로그는 추천 엔진이나 사용자 경험 개선에 활용된다.

## 🍡 데이터 소스와 클라이언트-서버 통신

- 데이터 소스는 데이터가 발생하는 출처이며, 대표적으로 서버, 웹, 앱이 있다.
- 클라이언트(예: 스마트폰, 웹브라우저)는 서버에 요청(request)을 보내고, 서버는 이에 대한 응답(response)을 전달하는 역할을 한다.

## 🍡 서버 로그와 클라이언트 로그의 차이

- 같은 행동이라도 서버와 클라이언트가 저장하는 로그는 다를 수 있다.
- 서버 로그는 처리 결과 중심의 정보를 저장하고, 클라이언트 로그는 UX 관점에서 클릭, 화면 진입 등 사용자의 실제 행동을 기록한다. 예를 들어, 로그인 전후 상태나 화면 뷰 등의 세부 행동은 클라이언트 로그에 포함될 수 있다.

| 로그 유형       | 저장 주체       | 주요 내용                     |
| --------------- | --------------- | ----------------------------- |
| 서버 로그       | 서버            | 요청 처리, 응답 상태 등       |
| 클라이언트 로그 | 사용자 디바이스 | 클릭, 화면 이동, 버튼 동작 등 |

## 🍡 데이터 저장소: 데이터베이스와 데이터 웨어하우스

- 데이터는 데이터베이스 또는 파일로 저장되게 되는데 서비스 운영에 사용되는 데이터베이스(RDB)는 빠른 검색과 트랜잭션 처리에 적합하지만, 분석에는 부적합할 수 있다.
- 때문에 분석 목적에는 OLAP 처리가 가능한 데이터 웨어하우스가 적합하며, Redshift, BigQuery, Snowflake가 대표적이며, 파일 저장은 서버 부담이 적지만 별도 처리가 필요합니다.

| 저장소            | 특징                        | 예시                |
| ----------------- | --------------------------- | ------------------- |
| 데이터베이스      | 빠른 검색, 수정 중심        | MySQL, PostgreSQL   |
| 데이터 웨어하우스 | 분석 특화, ETL 필요         | BigQuery, Snowflake |
| 파일              | 시스템 부담 낮음, 처리 필요 | CSV, JSON 등        |

## 🍡 운영 환경, 개발 환경, 데이터 타입

- 운영 환경(프로덕션)과 개발 환경(데브)은 분리되어야 하며, 대부분의 QA는 개발 환경에서 진행되어야 한다.
- 로그가 잘 작동하는지 확인한 후 운영 환경에서 적용하며, 메모리 효율을 위해 데이터 타입을 적절히 선택해서, 활용해야 한다.
- 데이터 타입에는 문자열(string), 정수(int), 실수(float), 타임스탬프(timestamp), 날짜(datetime) 등이 있고 특히 UTC 기준의 타임스탬프는 글로벌 서비스에서 중요하다.

| 데이터 타입 | 설명                                |
| ----------- | ----------------------------------- |
| string      | 문자열                              |
| int         | 정수                                |
| float       | 실수                                |
| timestamp   | 시간값 (1970년 1월 1일부터 초 단위) |
| datetime    | 날짜 (예: "2025-03-25")             |
